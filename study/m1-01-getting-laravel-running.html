02Why this concept exists: what is a backend?

You already know this machinery — I'm just naming it in web terms.

A web backend is nothing exotic. It is a long-running process that listens on a TCP port and speaks HTTP. It boots, binds a socket, and sits in a loop waiting for connections — the same socket-server shape you have written in C or Python.

And an HTTP request is, mentally, just a function call made over the network. Hover the pieces:

The browser makes the "call"; your backend is the "function" that runs and returns a value. That return value is an HTTP response: a status code (200 = ok, 404 = not found, …) plus a body (HTML, or JSON, or…).

The whole job, in one line

A backend is one loop: a request comes in → figure out what's being asked → do the work → send a response. Routing, controllers, models, auth — all just structure that keeps that loop organised as the app grows.

03What Laravel is (and what PHP is)

You could write that request→response loop by hand. For one endpoint, fine. For a real app — dozens of endpoints, auth, validation, a database — you would reinvent a mountain of plumbing.

Laravel is that plumbing, already built. It is a web framework: a body of code that runs the boring 90% (accept the connection, parse the HTTP request, match it to your code, help you talk to a database, format the response) so you write only the interesting 10% — the actual rules of your app.

Bridge — from what you know

Library vs. framework: a library is code you call; a framework is code that calls you. You'll write small pieces (a route handler, a model) and Laravel decides when to run them. That inversion of control is the whole idea of a framework.

Laravel is written in PHP. PHP is a C-family language (curly braces, semicolons) built specifically to generate web responses. It will feel familiar fast — the main visible quirk is that variable names start with $ ($user, $score). You will not write any PHP today; we pick up the syntax next page, when you write your first route.

04The four pieces you need

To run a Laravel app on a blank machine you need four things. Each one has an equivalent you already know:

PieceWhat it isYou already know it as
PHP The language runtime Laravel runs on the JVM / the Python interpreter
Composer PHP's dependency manager + package registry pip / Maven / npm
Laravel installer A tool that scaffolds a fresh Laravel app create-react-app, django-admin startproject
The dev server A small HTTP server for local development Flask's app.run()

Installing PHP cleanly on Windows by hand is fiddly, and fighting it is exactly where people quit. So we won't. We'll use Laravel Herd — the official, free installer that bundles PHP + Composer + the Laravel installer in a single click.

05Setup — install the toolchain

This is the blank-machine part. Three steps:

  1. Install Laravel Herd. Go to herd.laravel.com, download the Windows installer, run it, and accept the defaults. The free version is everything you need — ignore "Herd Pro". When it finishes, Herd is running (look for its icon in the system tray). It just put php, composer and laravel onto your system PATH.
  2. Open a fresh terminal. PATH changes only reach terminals opened after the install. Close every PowerShell window you have open, then open a new one. Skipping this is the #1 reason the next step "mysteriously" fails.
  3. Verify the three tools. Run each command below. You want a version number back, not an error.

Here is a safe practice terminal — it behaves like the real thing. Click the php --version chip, then try the others. (It also previews the commands from the next two sections.)

The real output will look roughly like this (your version numbers will differ — newer is fine):

expected output 
PHP 8.4.x (cli) ...
Composer version 2.x.x ...
Laravel Installer 5.x.x
Note

laravel --version shows the installer's version. The framework version of an app you create, you check later from inside the app. Don't expect the two numbers to match.

If a command says "not recognized"

You almost certainly skipped step 2 — open a brand-new terminal and retry. Still broken after a fresh terminal? Stop and come back to me. Don't burn an hour on it — getting unstuck fast is the real skill.

06Create the Laravel app

Your code lives in builder/ (these teaching pages live in study/). So create the app inside builder/:

powershell 
cd C:\Users\alexg\studying_for_hack_the_box\builder
laravel new backend

The installer asks a few setup questions. Exact wording shifts between versions, so match on the topic — here is what to choose and why:

If it asks about…Choose…Why
a starter kit None / no starter kit Starter kits scaffold login & UI for you. You're here to build that yourself — and our Vue frontend will be its own separate app, not a bundled one.
a testing framework Pest (just press Enter) Doesn't matter to us right now; take the default.
a database SQLite SQLite is just a file — zero setup, can't block us. We'll install a real PostgreSQL/MySQL database properly at Milestone 3, when challenges actually need one.
running migrations Yes "Migrations" is an M3 topic — just say yes. It only means "create the starter tables". Ignore the word for now.

When it finishes you'll have a new folder, builder/backend/, full of files.

Fallback

If php and composer work but laravel is somehow not found, you can create the app with composer create-project laravel/laravel backend instead — it does the same job.

Heads-up — not a problem

The installer creates its own git repo inside builder/backend/, so you'll have a git repo nested inside your main project repo. That's harmless for now — just know it's there. We'll decide how to handle git for the whole project later; don't touch it.

07Run it

powershell 
cd backend
php artisan serve

artisan is Laravel's command-line tool — it ships inside every Laravel app and you'll use it constantly. serve is one of its commands; it starts the local dev server. You'll see something like:

expected output 
   INFO  Server running on [http://127.0.0.1:8000].

   Press Ctrl+C to stop the server

Open the address it printed (usually http://127.0.0.1:8000) in your browser. You should see the Laravel welcome screen. That page was just generated by your own machine and sent over HTTP to your browser.

That URL is worth dissecting — every part of it is something you already know:

Two things to internalise

This terminal is now busy. The server runs in the foreground and holds the terminal until you stop it. To run other commands, open a second terminal. To stop the server, press Ctrl + C in this one.

Nothing here is new networking. 127.0.0.1 is the loopback address; 8000 is the TCP port the server bound. Same socket model as any server you've written.

08What actually just happened

When you loaded that page, here is the chain of events — five steps, all on your own machine:

1
You open the URL
The browser opens a TCP connection to 127.0.0.1:8000 and sends GET /.
2
The dev server receives it
The PHP process started by php artisan serve accepts the connection and hands the request to Laravel.
3
Laravel routes it
Laravel looks for a route matching / and finds the default welcome route.
4
Laravel builds the response
It generates the welcome page's HTML in memory and returns a 200 OK response.
5
The browser paints it
The HTML travels back over HTTP; the browser parses it and shows you the page.

The welcome page is not a file your browser opened. It is HTML built on the fly by a running PHP process and handed back over HTTP. Kill that process and the page is gone. That liveness is the backend — so prove it to yourself. Stop the server, then reload the browser, and watch what happens:
